# 桌面端完整首版设计

日期：2026-03-28

## 1. 概述

当前项目已经具备真实 TCP 传输链路、任务模型、设备发现、信任关系和浏览器受限访问，但桌面端仍是最小原型，缺少把这些能力组织成可用应用界面的交互层。下一轮需要把现有 `tkinter` 桌面壳扩展成完整首版桌面 UI，使用户可以通过图形界面完成设备选择、发送任务、接收确认、查看进度、查看历史记录和修改基础设置。

本轮保持现有 Python 标准库路线，不切换桌面技术栈，也不进入托盘、通知或安装包范围。

## 2. 目标与非目标

### 目标

- 提供完整的桌面主界面，覆盖设备列表、任务面板、历史记录和设置
- 支持通过界面选择目标设备并发送文件或文件夹
- 支持接收确认与拒绝
- 支持默认冲突策略选择
- 支持重试可恢复失败任务
- 支持拖拽发送入口

### 非目标

- 本轮不实现托盘、系统通知和后台最小化
- 本轮不切换到新的 GUI 技术栈
- 本轮不做安装包与跨平台发布流程
- 本轮不做复杂视觉打磨或高级主题系统
- 本轮不追求跨平台拖拽体验的极限一致性

## 3. 界面结构

桌面主界面分为四个区域。

### 3.1 设备列表区

用于展示：

- 自动发现的局域网设备
- 手动添加的设备
- 信任状态
- 当前选中目标设备

操作包括：

- 选择目标设备
- 手动输入地址添加设备
- 发起配对
- 刷新设备列表

### 3.2 任务面板区

用于展示当前活跃任务：

- 等待接收确认
- 传输中
- 失败可重试
- 已完成

每条任务至少显示：

- 目标设备
- 文件数量
- 已传输字节与总字节
- 当前状态
- 可用操作

### 3.3 历史记录区

用于展示最近完成或失败的发送与接收任务，与活跃任务面板分离，避免状态混杂。

### 3.4 设置区

只放本轮必须的配置：

- 本地共享目录
- 默认冲突策略
- 是否自动接收已信任设备任务
- 手动连接默认端口或地址信息

## 4. 模块边界

建议把现有 `DesktopShell` 扩展为以下职责分层。

### 4.1 DesktopAppState

负责从核心服务中读取设备、任务、历史和设置，并整理成界面模型。桌面组件不直接操作底层持久化结构。

### 4.2 DevicePanel

负责设备列表、目标设备选择、配对入口、手动添加设备和刷新操作。

### 4.3 TaskPanel

负责活跃任务展示、重试按钮、当前进度刷新和发送入口。

### 4.4 HistoryPanel

负责历史记录展示，只读，不承担复杂操作。

### 4.5 SettingsPanel

负责基础设置读取与更新，包括默认冲突策略和共享目录。

## 5. 交互流

### 5.1 发送流程

用户先在设备列表中选择目标设备，然后通过：

- 选择文件
- 选择文件夹
- 拖拽到主窗口

来创建发送任务。

若未选择目标设备，则界面提示必须先选定目标，而不是自动猜测发送对象。

### 5.2 接收确认流程

当收到新的传输任务时，桌面端弹出确认对话框，展示：

- 发送方名称
- 文件数量
- 默认接收目录
- 默认冲突策略

用户可以：

- 接受
- 拒绝
- 调整本次任务的冲突策略

### 5.3 冲突处理流程

首版将冲突处理设为任务级默认策略，而不是每个文件逐个交互。用户在设置区配置默认值，也可以在接收确认时覆盖本次任务的策略。

支持策略：

- 覆盖
- 跳过
- 重命名

### 5.4 重试流程

对 `retryable` 任务，任务面板显示“重试”按钮。用户点击后直接复用已有任务 ID 和恢复状态重新发起，不要求重新选文件。

### 5.5 历史记录浏览

历史记录区只读展示最近任务，帮助用户区分当前任务与已结束任务，不在历史记录区重复放置发送控制。

## 6. 错误处理

桌面端需要显式区分并展示以下错误或状态：

- 未选择目标设备就尝试发送
- 目标设备离线或连接失败
- 配对未完成导致无法发送
- 接收方拒绝任务
- 任务失败但可重试
- 冲突策略导致部分文件跳过
- 手动地址无效
- 共享目录不可写

错误信息应在任务面板、状态栏或确认对话框中以明确语义展示，而不是只输出到日志。

## 7. 测试策略

### 7.1 状态单元测试

验证：

- 设备列表映射
- 当前任务视图状态
- 历史记录排序与展示
- 设置更新行为
- 可重试任务按钮状态

### 7.2 UI 交互测试

验证：

- 选中设备后发送文件或文件夹
- 接收确认接受与拒绝
- 默认冲突策略更新
- `retryable` 任务重试

### 7.3 集成回归测试

验证桌面层接入后没有破坏：

- TCP 真实传输
- 信任关系
- 浏览器受限访问

## 8. 技术约束

本轮继续使用 `tkinter`，尽量基于现有桌面壳扩展，而不是重写为新技术栈。为了保持跨平台可运行性，界面实现优先采用标准控件和明确布局，不依赖第三方主题库。

拖拽发送允许采用“主窗口接收拖入路径文本或平台可用的最小拖拽集成”作为首版实现，只要行为清晰可用即可。
