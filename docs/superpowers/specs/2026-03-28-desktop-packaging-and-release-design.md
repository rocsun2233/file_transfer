# 桌面打包与发布设计

日期：2026-03-28

## 1. 概述

当前项目已经具备真实 TCP 传输链路和完整首版桌面 UI，但仍缺少统一的桌面分发方式。下一轮需要为 Windows、Linux、macOS 补齐构建脚本、产物目录、版本信息和发布说明，使当前程序能够进入可分发的预览版状态。

本轮仍以桌面端为主，同时在发布说明中明确预留 Android 浏览器接入方向，但不提供 Android 安装包或原生客户端。

## 2. 目标与非目标

### 目标

- 为 Windows、Linux、macOS 提供统一的打包与发布目录结构
- 使用 `PyInstaller` 生成三端单目录分发产物
- 提供统一版本信息、构建清单和快速启动说明
- 补充桌面产物级验证流程
- 在发布说明中明确 Android 暂时通过浏览器接入的预留说明

### 非目标

- 本轮不实现 Android 原生客户端
- 本轮不做 Android Web 页面优化
- 本轮不做签名、公证、商店发布或自动更新
- 本轮不做托盘、通知等额外桌面行为
- 本轮不要求在单一环境内完成三端实际构建

## 3. 产物结构

建议统一产物结构如下：

- `dist/windows/HybridTransfer/`
- `dist/linux/HybridTransfer/`
- `dist/macos/HybridTransfer/`

每个平台目录内包含：

- 主程序入口
- 运行所需依赖文件
- 快速启动说明
- 版本信息或构建元数据

仓库级发布信息放在：

- `release/manifest.json`
- `release/README.md`
- `release/CHANGELOG.md`

## 4. 构建方式

### 4.1 打包方式

采用 `PyInstaller` 单目录模式。该模式更适合当前 Python 程序处理资源、配置目录和跨平台差异，不强求将所有内容挤入单个可执行文件。

### 4.2 构建脚本

建议新增：

- `scripts/build_windows.sh`
- `scripts/build_linux.sh`
- `scripts/build_macos.sh`

如有需要，可补充 Windows 本地使用的批处理脚本。

构建脚本负责：

- 清理旧产物
- 执行 `PyInstaller`
- 整理输出目录
- 复制快速启动说明
- 生成或复制发布元数据

## 5. 版本与发布元数据

首版建议采用简单语义版本，例如 `0.1.0`。

发布元数据至少包含：

- 版本号
- 构建日期
- 支持平台
- 当前已实现能力摘要
- 已知限制
- Android 当前接入方式说明

## 6. Android 预留说明

本轮发布说明必须明确：

- 当前只提供 Windows、Linux、macOS 的桌面分发产物
- Android 当前通过浏览器访问接入
- Android 原生客户端未实现
- 移动端优化将在后续 change 中推进

这样可以避免用户把桌面发布成果误解成已支持 Android 本地安装。

## 7. 错误处理

构建和发布流程至少要明确以下失败类型：

- `PyInstaller` 未安装
- 构建脚本依赖缺失
- 入口脚本路径错误
- 资源复制失败
- 产物目录整理失败
- 版本元数据缺失
- 快速启动说明缺失
- 打包后程序启动时默认共享目录无法创建

构建脚本应对这些情况输出明确错误信息并以非零状态退出。

## 8. 验证策略

### 8.1 代码级验证

至少执行：

- 自动化测试
- 语法检查

确保进入打包流程的是可运行代码。

### 8.2 产物级验证

每个平台构建后至少验证：

- 程序可启动
- 主窗口可打开
- 默认共享目录可创建
- TCP 传输服务可启动
- Web 入口可启动
- 版本与说明文件存在

## 9. 技术约束

当前项目仍是 Python 桌面程序，因此三端构建默认在各自平台本机执行，不假设单机交叉编译。构建流程要尽量使用简单、可读、可重复执行的脚本，而不是依赖隐式手工步骤。

## 10. 后续衔接

本轮完成后，后续可以继续推进：

- Android 浏览器访问体验优化
- Android 原生客户端可行性评估
- 桌面端签名与公证
- 自动更新机制
- 更正式的安装包形态
