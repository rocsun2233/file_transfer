# 局域网文件传输小程序设计

日期：2026-03-28

## 1. 概述

本项目要构建一个跨平台文件传输小程序，运行于 Windows、Linux、macOS。
产品采用混合方案：安装客户端的设备使用桌面应用完成设备发现、配对、任务管理和稳定传输；未安装客户端的设备可通过浏览器访问在线设备暴露的本地 Web 页面，完成受限的上传和下载操作。

首版目标是覆盖局域网内的增强协作场景，包含：

- 自动发现局域网在线设备
- 首次连接确认或配对码建立信任
- 点对点文件与文件夹传输
- 传输进度显示
- 历史记录
- 断点续传
- 批量任务
- 拖拽发送
- 浏览器上传与下载入口

首版不包含互联网穿透、账号体系、云同步、长期分享链接和跨网络远程控制，但架构必须为后续互联网穿透保留扩展接口。

## 2. 目标与非目标

### 目标

- 在同一局域网内为三大桌面平台提供统一的文件传输体验
- 兼顾已安装客户端设备与临时浏览器访问设备
- 在可信网络环境下提供平衡型安全模型
- 通过任务会话模型支持断点续传、批量任务和后续链路切换
- 为未来公网发现、NAT 打洞和中继传输预留清晰接口

### 非目标

- 首版不提供互联网穿透能力
- 首版不做账号登录或云端身份系统
- 首版不做移动端原生应用
- 首版不提供端到端加密方案以外的复杂企业权限管理

## 3. 产品形态

系统由桌面主应用和内置 Web 入口组成。

- 桌面应用是主要交互入口，负责设备列表、发送任务、拖拽发送、历史记录、通知和设置。
- 每台设备的本地核心服务对外暴露一个 Web 页面，供未安装客户端的设备通过浏览器临时访问。
- 浏览器入口提供轻量能力，不承担复杂任务编排；复杂操作仍由桌面端完成。

## 4. 安全模型

安全策略采用平衡型方案：

- 设备默认在局域网内可被发现
- 首次连接必须通过一次性确认或配对码建立信任
- 建立信任后，同一设备后续访问默认免确认
- 用户可在设置中撤销设备信任关系
- 未配对浏览器访问仅能进入受限入口，需要验证码或桌面端确认后才能上传下载

此模型适合家庭或办公室可信局域网，避免完全裸露访问，同时不引入账号体系复杂度。

## 5. 系统架构

系统采用三层架构。

### 5.1 桌面壳层

负责：

- 托盘与主窗口
- 设备列表展示
- 拖拽发送
- 传输任务面板
- 历史记录视图
- 系统通知
- 本地设置管理

桌面层不直接处理网络发现和传输协议，只调用本地核心服务公开的接口。

### 5.2 本地核心服务层

这是每台设备上的常驻进程，负责：

- 设备发现
- 配对认证
- 信任关系管理
- 任务调度
- 传输会话管理
- 断点续传
- 历史记录存储
- 冲突处理
- 内置 Web 服务

桌面 UI 和浏览器页面都作为该服务的客户端。

### 5.3 传输与连接抽象层

本层定义统一接口，使业务层不依赖具体链路实现。

首版实现：

- `LanDiscoveryAdapter`：基于局域网广播或组播发现设备
- `DirectTransferAdapter`：局域网内直接建立传输通道
- `LocalWebGateway`：为浏览器访问提供本地 HTTP 入口

后续扩展：

- `CloudDirectoryAdapter`：用于跨公网设备定位
- `NatTraversalAdapter`：用于 NAT 打洞
- `RelayTransferAdapter`：在直连失败时回退到中继传输

业务层只关心发现、信任、任务和状态，不关心底层是局域网直连还是未来公网链路。

## 6. 核心数据流

### 6.1 设备发现

- 设备启动后在局域网中广播自身信息
- 同时监听其他设备广播并构建设备列表
- 若自动发现失败，可手动输入地址补充连接

### 6.2 首次配对

- 用户从设备列表中选择目标设备
- 发送方发起连接请求
- 接收方显示确认请求，或要求输入一次性配对码
- 成功后生成持久化信任关系，供后续免确认访问

### 6.3 文件传输

- 发送方创建任务会话，生成任务 ID
- 任务元数据包括文件清单、目录结构、大小、校验信息和目标设备
- 接收方确认后，双方建立数据通道并进行分片传输
- 进度、失败重试和断点续传均绑定到任务 ID

### 6.4 浏览器访问

- 已配对设备可通过浏览器打开目标设备的本地 Web 页面
- Web 页面支持受控浏览、上传、下载和查看最近任务
- 未配对访问进入受限入口，需要验证码或桌面端确认

## 7. 未来互联网穿透兼容设计

互联网穿透不是首版功能，但需在架构上可增量接入。

连接建立流程设计为：

1. 优先尝试局域网发现与直连
2. 若目标设备不在同一局域网，切换到云端目录定位
3. 尝试 NAT 打洞建立直连
4. 若打洞失败，回退到中继传输

对上层业务而言，仍然是同一个“发送到设备”的动作。
因此任务状态机、配对模型和传输会话抽象不能与局域网实现硬编码耦合。

## 8. 错误处理

系统需要区分以下错误或状态：

- 设备发现失败
- 首次配对被拒绝
- 连接超时
- 传输中断
- 断点恢复失败
- 文件冲突
- 浏览器访问权限不足
- 局域网直连失败
- 未来公网打洞失败
- 未来中继链路降级

文件冲突至少支持三种策略：

- 覆盖
- 跳过
- 重命名

任务必须支持：

- 暂停
- 继续
- 失败重试
- 中断后恢复

## 9. 测试策略

测试分三层。

### 9.1 单元测试

覆盖：

- 设备发现状态机
- 配对状态机
- 任务调度逻辑
- 断点续传逻辑
- 冲突处理策略

### 9.2 接口与协议测试

验证：

- 桌面 UI 与核心服务接口契约一致
- Web 页面与核心服务接口契约一致
- 传输适配器遵守统一会话接口

### 9.3 跨平台集成测试

至少验证以下场景：

- Windows 到 Linux
- Linux 到 macOS
- macOS 到 Windows
- 文件传输
- 文件夹传输
- 批量任务
- 断点恢复
- 浏览器上传下载

即使首版不实现互联网穿透，也应对未来适配器提供替身或模拟测试入口，防止后续接入新链路时破坏现有任务模型。

## 10. 技术选型建议

建议技术栈：

- 桌面端：Tauri + TypeScript
- 核心服务：Rust
- Web 页面：React
- 本地存储：SQLite
- 局域网发现：mDNS 与 UDP 广播双通道兼容设计
- 数据传输：基于统一会话抽象的 TCP/QUIC 可替换传输适配器

选型理由：

- Rust 适合实现跨平台、高可靠、常驻的传输内核
- Tauri 能以较轻体量覆盖三大桌面平台
- Web 页面可复用前端能力，为未安装客户端设备提供低门槛访问
- SQLite 足以支撑本地信任关系、历史记录和任务元数据
- 发现与传输抽象便于未来扩展互联网穿透和中继能力
