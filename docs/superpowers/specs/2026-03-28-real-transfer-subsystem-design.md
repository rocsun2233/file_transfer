# 真实传输子系统设计

日期：2026-03-28

## 1. 概述

当前项目已经具备局域网发现、配对、任务模型、浏览器访问和最小桌面壳，但真实设备到设备的数据传输仍停留在原型层。下一轮需要把现有任务模型补成真正可在局域网内运行的 TCP 文件传输子系统，使两台安装客户端的设备能够完成文件、文件夹和批量任务的实际发送、接收、恢复和冲突处理。

本轮只补桌面设备到桌面设备的主传输链路，不改变浏览器入口的次级定位，也不进入互联网穿透或安装包发布范围。

## 2. 目标与非目标

### 目标

- 让两台局域网内的桌面客户端能够实际传输文件数据
- 支持文件、文件夹和批量任务
- 支持接收方任务确认
- 支持断点续传
- 支持覆盖、跳过、重命名三种冲突处理
- 支持失败重试并与现有任务状态持久化打通

### 非目标

- 本轮不实现互联网穿透
- 本轮不让浏览器端承担主传输链路
- 本轮不做完整桌面 UI 打磨
- 本轮不做安装包与发布流程
- 本轮不做并发多连接、带宽控制等性能优化

## 3. 模块边界

本轮在现有核心服务中新增真实传输子系统，核心模块如下。

### 3.1 TransferCoordinator

负责从任务模型出发驱动真实传输流程：

- 扫描待发送文件与文件夹
- 生成任务清单
- 发起任务握手
- 推进文件与分片发送
- 写回任务进度
- 处理中断恢复和失败重试

该模块是任务层与传输协议层之间的桥梁，不直接面向 UI。

### 3.2 TransferProtocol

定义应用层协议消息格式和状态流转，至少包含：

- `TASK_OFFER`
- `TASK_ACCEPT`
- `TASK_REJECT`
- `CHUNK`
- `CHUNK_ACK`
- `RESUME_REQUEST`
- `RESUME_STATE`
- `TASK_COMPLETE`
- `TASK_ERROR`

该模块只负责协议帧和消息语义，不负责磁盘写入或桌面展示。

### 3.3 TcpTransferServer 与 TcpTransferClient

负责：

- 建立 TCP 监听与连接
- 收发协议消息
- 分片发送与接收
- 临时文件写入
- 文件完成后原子落盘

这两部分构成首版局域网真实传输的底层数据通道。

### 3.4 ResumeIndex

负责持久化记录：

- 任务 ID
- 文件相对路径
- 已确认分片
- 临时文件位置
- 最后更新时间
- 恢复所需元数据

它是断点续传和失败重试的基础。

## 4. 传输数据流

### 4.1 任务发起

发送端从当前选择的文件和文件夹生成任务计划，包含：

- 任务 ID
- 目标设备 ID
- 文件清单
- 相对路径
- 文件大小
- 修改时间
- 可选校验值

发送端通过 TCP 建立控制连接后发送 `TASK_OFFER`。

### 4.2 接收确认

接收端收到 `TASK_OFFER` 后先验证发送方是否可信。

- 若未建立信任关系，则进入配对确认流程
- 若已建立信任关系，则进入任务确认流程

任务确认包含：

- 是否接受本次任务
- 接收根目录
- 默认冲突策略
- 是否存在可恢复的旧任务状态

接受后返回 `TASK_ACCEPT`；拒绝则返回 `TASK_REJECT`。

### 4.3 分片传输

每个文件按固定大小拆分为分片。发送端顺序发送 `CHUNK`，接收端写入临时文件并返回 `CHUNK_ACK`。

首版采用保守策略：

- 文件级顺序传输
- 分片级确认
- 不启用复杂滑动窗口

这样可以降低协议复杂度，先保证正确性和可恢复性。

### 4.4 断点续传

传输中断后，发送端重连并发送 `RESUME_REQUEST`。接收端查询 `ResumeIndex`，返回当前任务已完成的文件与分片信息。发送端仅重发缺失分片。

文件写入采用临时文件策略，只有文件完整且校验通过后才移动到目标位置，避免中断后留下伪完成文件。

### 4.5 冲突处理

若目标路径已存在，接收端按用户选择执行：

- `overwrite`
- `skip`
- `rename`

冲突结果必须回传给发送端，使双方对任务中文件最终落点保持一致。

### 4.6 任务完成

所有文件完成后，接收端返回 `TASK_COMPLETE`。发送端收到后将任务标记为完成并写入历史记录。

若任意步骤失败，则返回 `TASK_ERROR` 并记录失败原因与恢复位置。

## 5. 错误处理

本轮必须明确区分以下状态或错误：

- TCP 连接建立失败
- 对端拒绝任务
- 信任校验失败
- 单文件冲突
- 分片写入失败
- 校验失败
- 连接中断
- 恢复失败
- 用户取消任务

任务状态机至少包含：

- `pending`
- `awaiting_accept`
- `in_progress`
- `paused`
- `failed`
- `completed`

## 6. 测试策略

### 6.1 协议单元测试

验证协议消息编解码、任务流转和错误消息处理：

- `TASK_OFFER`
- `TASK_ACCEPT`
- `CHUNK`
- `CHUNK_ACK`
- `RESUME_REQUEST`
- `TASK_COMPLETE`

### 6.2 传输集成测试

基于本机 loopback 启动发送端与接收端，验证：

- 单文件传输
- 文件夹传输
- 批量任务
- 冲突处理
- 断点续传
- 失败重试

### 6.3 端到端回归测试

验证新传输子系统接入后没有破坏：

- 设备信任关系
- 任务历史记录
- 浏览器受限访问能力

## 7. 技术约束

本轮优先使用 Python 标准库与现有代码结构完成，保持当前原型连续性。底层连接基于 TCP，自定义应用层协议帧；协议消息优先使用易调试的文本或长度前缀 JSON 表达，文件数据用显式长度帧传输。

不引入新的重量级依赖，除非实现关键路径时发现标准库无法满足最小可用要求。
