## Context

当前项目已经具备真实 TCP 传输链路、任务状态机、局域网发现、信任关系和浏览器受限访问，但桌面端只有一个最小 `tkinter` 壳，主要用于手动添加设备和查看历史记录。它还没有把底层能力组织成完整的桌面工作流，无法支撑一个接近首版可用性的局域网传输应用。

这一轮需要继续沿用 `tkinter` 和 Python 标准库，把现有原型扩展成完整首版桌面主界面，覆盖设备选择、发送任务、接收确认、重试入口、历史记录和基础设置，而不引入新的 GUI 技术栈。

## Goals / Non-Goals

**Goals:**
- 提供包含设备列表、任务面板、历史记录区和设置区的完整桌面主界面
- 支持通过图形界面发送文件、发送文件夹和拖拽发送入口
- 支持接收确认与拒绝
- 支持默认冲突策略展示和任务级覆盖
- 支持对 `retryable` 任务执行重试
- 让桌面组件通过统一状态层消费核心服务数据

**Non-Goals:**
- 本轮不实现托盘、系统通知和后台最小化
- 本轮不切换到其他 GUI 框架
- 本轮不做安装包和发布流程
- 本轮不做复杂主题系统或视觉精修

## Decisions

### 1. 继续扩展现有 tkinter 桌面壳，而不是重写技术栈

这轮主要目标是把已有底层能力变成可操作界面，而不是换 UI 技术。继续扩展现有 `tkinter` 结构可以最小化工程风险，保持当前代码连续性。

备选方案是改为本地 Web UI 或引入其他 Python GUI 框架。前者会把桌面和浏览器入口重新缠在一起，后者会引入新依赖和新工程面，因此都不采用。

### 2. 使用 DesktopAppState 作为桌面状态整理层

桌面组件不直接读取底层持久化字典，而是通过一个专门的状态整理层获得设备、任务、历史记录和设置的界面模型。这样可以让 UI 更新逻辑和核心服务结构解耦。

备选方案是让每个面板直接访问核心服务。这样短期实现更快，但会让状态访问分散且难以测试，因此不采用。

### 3. 将主界面拆为设备、任务、历史和设置四个面板

这种结构与产品主流程一致：先选设备，再看任务，再回看历史，并在设置区调整默认行为。它也适合渐进式扩展，不需要一次重写全部桌面代码。

### 4. 接收确认在桌面端用明确对话框承载

当有新的传输任务到达时，桌面端通过对话框展示发送方、文件数量、目标目录和默认冲突策略，并允许用户接受、拒绝或覆盖本次任务策略。这样既能体现已有的底层接受/拒绝能力，又不会把任务确认藏在日志或后台状态里。

### 5. 重试入口只放在任务面板

`retryable` 任务由任务面板提供重试按钮，复用原有任务 ID 和恢复状态。历史记录区保持只读，避免“当前控制”和“已完成浏览”职责混杂。

## Risks / Trade-offs

- [tkinter 的布局与交互能力有限] → 先优先保证信息结构清晰和行为可用，不追求复杂视觉表现。
- [桌面状态与底层任务状态可能失配] → 通过单独的 `DesktopAppState` 统一映射，避免多组件自行解释底层数据。
- [拖拽跨平台细节差异较大] → 首版允许用最小可用的拖拽入口，不把高级平台一致性作为本轮目标。
- [接收确认可能阻塞主界面] → 用简洁确认对话框承载，并限制确认内容在任务级而不是文件级。
- [设置项与运行时行为容易脱节] → 只开放少量关键设置，并在更新后立即回写核心服务可见状态。

## Migration Plan

无需迁移外部系统，但需要对现有桌面壳做增量演进：

1. 引入桌面状态层和面板划分
2. 把设备列表、任务面板、历史记录和设置区布局到主窗口
3. 接入发送入口、接收确认和重试入口
4. 接入基础设置读写与冲突策略更新
5. 增加状态测试、交互测试和回归验证

若拖拽入口在某个平台表现不稳定，可先保留“选择文件/文件夹”入口，不影响总体界面结构。

## Open Questions

- 首版拖拽入口是否采用纯窗口拖入提示，还是需要额外的显式拖拽区域。
- 历史记录区是内嵌在主窗口下半部分，还是独立为标签页更合适。
- 设置更新后是否立即持久化，还是等待用户显式点击保存。
