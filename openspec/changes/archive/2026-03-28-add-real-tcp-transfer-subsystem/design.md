## Context

当前实现已经具备局域网发现、配对、任务状态持久化、浏览器受限访问和最小桌面壳，但真正的桌面到桌面文件数据传输还没有完成。现有 `TaskManager` 主要记录任务创建、进度和历史，尚未驱动真实文件分片发送、接收确认、恢复或失败重试。

这一轮需要在保持 Python 标准库路线和现有项目结构连续性的前提下，为局域网直连补充一个最小但可靠的 TCP 传输子系统。该子系统要与现有信任模型和任务历史打通，同时为后续桌面 UI 完整化和互联网穿透适配保留边界。

## Goals / Non-Goals

**Goals:**
- 让两台局域网内的桌面客户端能够通过真实 TCP 数据通道传输文件
- 支持文件、文件夹和批量任务
- 支持接收确认、任务拒绝、失败重试和断点续传
- 将真实传输结果写回现有任务状态与历史记录
- 保持协议层、协调层和磁盘写入层边界清晰

**Non-Goals:**
- 本轮不实现互联网穿透
- 本轮不让浏览器入口承担主传输链路
- 本轮不做完整桌面 UI 打磨
- 本轮不做多连接并发、带宽限制等高级优化

## Decisions

### 1. 使用 TransferCoordinator 驱动任务到传输的衔接

`TransferCoordinator` 负责把现有任务模型转为真实传输过程，包括文件扫描、任务握手、分片发送、确认处理、恢复和失败重试。这样可以保留 `TaskManager` 作为状态存储层，而不是把网络与文件系统逻辑直接塞进状态模块。

备选方案是把真实传输逻辑直接写入 `TaskManager`。这种方式会让状态存储和副作用执行耦合，后续难以测试和扩展，因此不采用。

### 2. 使用 TCP + 应用层分片协议，而不是 HTTP 复用

真实传输链路基于 TCP，应用层定义 `TASK_OFFER`、`TASK_ACCEPT`、`CHUNK`、`CHUNK_ACK`、`RESUME_REQUEST`、`TASK_COMPLETE` 等消息。协议消息优先采用长度前缀 JSON 帧，文件数据使用显式长度的二进制分片帧。

备选方案是基于现有本地 Web 服务继续扩展 HTTP 上传下载。该方案对浏览器友好，但不适合作为桌面主链路承载接收确认、恢复和批量传输，因此不采用。

### 3. 首版采用保守的顺序分片确认模型

发送端按文件顺序处理，每个文件再按固定大小分片；接收端写入临时文件并对每个分片返回确认。只有收到确认后，发送端才推进后续分片。

这种方式吞吐量不是最优，但协议和恢复模型最清晰，适合先把正确性做稳。高级滑动窗口和并发多文件传输留到后续阶段。

### 4. 断点续传依赖 ResumeIndex 和临时文件落盘

接收端在本地记录已完成分片、临时文件位置和任务恢复元数据。连接中断后，发送端通过 `RESUME_REQUEST` 获取缺失分片，仅补发未确认部分。文件只有在完整且校验通过后才移动到最终路径。

备选方案是按文件级粒度恢复，整文件重发。它实现更简单，但对大文件和批量任务代价太高，因此不采用。

## Risks / Trade-offs

- [顺序分片确认降低吞吐量] → 先保证协议正确性和可恢复性，后续再考虑窗口优化。
- [协议与任务状态机衔接容易失配] → 用 `TransferCoordinator` 作为唯一协调入口，避免多处更新任务状态。
- [恢复索引损坏会导致续传失败] → 用明确的本地索引文件结构，并在恢复失败时回退到整文件重传。
- [接收确认增加交互阻塞] → 将任务确认限制在任务开始阶段，不把每个文件都做成人工确认。
- [标准库 TCP 实现需要更多自管细节] → 保持协议帧简单、测试覆盖完整，避免引入额外依赖。

## Migration Plan

无需迁移已有外部系统，但需要在当前原型内增量接入：

1. 新增协议模块、传输协调器和恢复索引
2. 新增 TCP 监听与客户端连接逻辑
3. 将真实传输过程接入现有任务状态与历史记录
4. 为冲突处理、失败重试和断点续传补充持久化字段
5. 增加 loopback 传输集成测试和回归验证

如接入后发现恢复逻辑不稳定，可先保留真实传输但临时降级为“失败后整文件重发”，不影响整体架构方向。

## Open Questions

- 首版每个分片的默认大小应采用固定值还是按文件大小分档。
- 文件完成校验是否默认使用哈希，还是首版只基于大小与分片确认。
- 接收方默认保存目录是独立共享目录，还是沿用当前任务上下文目录。
