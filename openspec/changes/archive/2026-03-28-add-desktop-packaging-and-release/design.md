## Context

当前项目已经从原型推进到具备真实传输和桌面工作台的阶段，但还没有统一的分发方式。没有构建脚本、发布元数据和产物结构，意味着当前程序仍然偏向开发环境使用，而不是“可交付的预览版桌面应用”。

这一轮需要在不改变核心传输行为的前提下，为 Windows、Linux、macOS 制定统一的打包与发布流程，并在发布信息中明确 Android 仍处于浏览器接入预留状态。

## Goals / Non-Goals

**Goals:**
- 提供基于 `PyInstaller` 的桌面三端打包方案
- 提供统一的 `dist/` 和 `release/` 目录结构
- 提供版本号、构建时间、支持平台和已知限制等发布元数据
- 提供快速启动说明和产物级验证流程
- 明确 Android 当前只通过浏览器接入

**Non-Goals:**
- 本轮不实现 Android 原生客户端
- 本轮不做 Android Web 页面优化
- 本轮不做签名、公证、商店发布或自动更新
- 本轮不要求单一环境内完成三端交叉编译

## Decisions

### 1. 采用 PyInstaller 单目录分发模式

当前项目是 Python 桌面程序，单目录模式更适合处理后续资源、配置目录和跨平台差异，也更便于问题排查。相比单文件模式，单目录模式更稳定、对运行资源更友好。

### 2. 平台分别构建，而不是假设交叉编译

Windows、Linux、macOS 各自用本机构建脚本打包，不把交叉编译当作本轮目标。这样可以避免为不同平台兼容性再引入额外复杂度。

### 3. 统一产物目录与发布目录职责

`dist/<platform>/HybridTransfer/` 用于放平台分发产物，`release/` 用于放跨平台共享的版本清单、README 和 changelog。这样可以区分“具体可运行包”和“发布说明”。

### 4. Android 仅在发布说明中作为预留方向

本轮不生成 Android 包，也不尝试把桌面产物复用到 Android。发布元数据中只明确说明 Android 当前通过浏览器接入，避免误导用户。

## Risks / Trade-offs

- [三端无法在当前单一环境全部实际构建] → 用平台独立脚本和清晰验证步骤保证可重复执行。
- [PyInstaller 在不同平台行为有差异] → 每个平台各自整理脚本和说明，不强求完全相同的细节实现。
- [发布文档与实际产物脱节] → 让构建脚本在打包时同步复制或生成元数据。
- [用户误解 Android 已支持原生安装] → 在 README 和 manifest 中明确写出 Android 仅通过浏览器接入。
- [打包前代码质量不足会放大分发问题] → 构建流程前置自动化测试和语法检查。

## Migration Plan

这轮无需迁移现有业务行为，但要增量补充交付能力：

1. 新增打包依赖说明与构建脚本
2. 新增 `dist/` 与 `release/` 目录约定
3. 新增版本元数据和快速启动说明模板
4. 新增产物级验证脚本或检查步骤
5. 在文档中写明 Android 预留边界

如果某个平台本轮无法在当前环境内实机验证，仍应保留明确脚本和验证清单，以便在对应平台补跑。

## Open Questions

- 版本号是写死在单独文件中，还是由构建脚本从单一版本源生成。
- Windows 是否需要单独的 `.bat` 启动辅助脚本。
- macOS 首版是否只交付 PyInstaller 产物目录，还是额外整理为 `.app` 目录结构。
