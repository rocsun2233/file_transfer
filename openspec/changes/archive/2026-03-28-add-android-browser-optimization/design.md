## Context

当前项目已经有本地 Web 入口和桌面主链路，但浏览器入口主要满足桌面或通用访问，没有针对 Android 手机浏览器的小屏和触控体验做专门优化。用户在 Android 上访问时，最需要的是快速知道“当前有无权限访问、是否在等待桌面确认、现在能否上传下载”，而不是桌面式的信息密度。

这一轮继续沿用现有本地 Web 入口和后端逻辑，不单独创建移动站点或 Android 原生客户端，而是在当前服务端视图输出上增加移动端状态模型和更适合手机的页面结构。

## Goals / Non-Goals

**Goals:**
- 为 Android 浏览器提供移动优化的状态页和操作入口
- 支持触控友好的上传、下载、刷新和最近任务查看
- 明确展示等待确认、验证码错误、权限拒绝和会话过期等状态
- 保持与现有桌面浏览器入口共用后端逻辑

**Non-Goals:**
- 本轮不实现 Android 原生客户端
- 本轮不实现 PWA 安装能力
- 本轮不做复杂任务管理或离线支持
- 本轮不引入新的前端框架

## Decisions

### 1. 继续扩展现有本地 Web 入口，而不是新建移动站点

这样可以复用现有权限判断、上传下载和最近任务数据，不会把桌面浏览器入口与移动入口拆成两套后端行为。

### 2. 用 MobileWebViewModel 组织移动端状态和展示信息

移动浏览器需要更强的状态化展示，因此通过单独的移动视图模型来表达“当前能不能传、为什么不能传、最近发生了什么”，避免直接把桌面视图结构硬塞到手机上。

### 3. 页面结构以状态优先、操作其次

Android 访问首先要看到权限和会话状态，因此页面顶部优先展示访问状态，而不是直接列文件。上传、下载和刷新放在操作区，最近任务和连接说明次之。

### 4. 错误反馈使用短句状态块

移动端用户需要立即理解错误类型，因此用明确短句展示权限错误、连接错误和操作错误，而不是通用 HTTP 错页或技术性信息。

## Risks / Trade-offs

- [继续复用同一套页面可能保留桌面遗留结构] → 通过移动视图模型和分区输出降低桌面布局痕迹。
- [不引入前端框架会限制交互丰富度] → 本轮优先保证清晰和可用，而不是追求复杂前端能力。
- [移动端状态过多会让页面复杂] → 用“状态优先、操作其次”的顺序收敛信息。
- [Android 浏览器行为存在差异] → 首版聚焦文件选择、下载和状态反馈等共性能力，不做更深平台绑定。

## Migration Plan

无需迁移现有桌面链路，但要对当前 Web 入口做增量增强：

1. 新增移动端视图模型
2. 调整本地 Web 页面输出结构
3. 将权限状态、最近任务和错误反馈移动端化
4. 增加针对 Android 浏览器访问的测试
5. 回归验证桌面浏览器访问和桌面主链路没有被破坏

## Open Questions

- 移动端页面是否需要根据 User-Agent 自动切换，还是统一用响应式视图模型输出。
- 最近任务是否只显示最近几条，还是显示完整历史入口。
- 当前下载入口在 Android 浏览器中是否需要更明确的“下载开始”反馈文案。
